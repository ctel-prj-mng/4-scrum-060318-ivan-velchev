Канбан
метод управления разработкой, реализующий принцип «точно в срок» и способствующий равномерному распределению нагрузки между работниками. При данном подходе весь процесс разработки прозрачен для всех членов команды. Задачи по мере поступления заносятся в отдельный список, откуда каждый разработчик может извлечь требуемую задачу.

История термина

Этот термин пришёл к нам из Японии благодаря широко известной в узких кругах производственной системе Тойота.

Основопологающие принципы

  1. Опора на существующие методы разработки;
  2. Преждевременая договорёность о внесении изменений;
  3. Уважение к существующему порядку, ролям и обязаностям;
  4. Поощьрение инищиативы.

Правила канбан

  1. Перед разработчиком стоит определёный
    план.Данный план может быть сортирован по
    приоритету и может быть изменён в любое время;
  2. Количество задач стоящих перед
    разработчиками одновремено ограничено.

Сфера применения

Методология канбан плохо работает для продуктовых
команд, но отлично подходит для
таких команд как:

  1. Группы тестирования, работающие отдельно от групп разработки;
  2. Службы поддержки;
  3. Неосновные производства.

Сфера применения

Отдельно необходимо отметить, что
методология Канбан хорошо работает в
стартапах, не имеющих чёткого плана, но
активно работающих над разработкой.

Ограничение методологии

  1. Методология плохо работает с большими командами;
  2. Не подходит для работы с кросс-функциональными командами;
  3. Не предназначена для долгосрочного планирования.

Отличие методологии Канбан от других методологий

  1. Отсуцтвия строгих временых рамок;
  2. Меньшее количество задач, но они более объёмные;
  3. Скорость работы каманды отсутствкет и считается только средние время на полную реализацию задачи.

Канбан разработка в первую очередь орентирована на выполнение задачи.

Реализация задачи делается только когда, когда задача выполнена.
